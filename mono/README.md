# MONO — ミニマリズム持ち物管理 PWA

写真またはバーコードから持ち物を登録し、「今何を持っているか」「増えすぎていないか」を
ひと目で把握するための、自分専用・スマホ向け PWA です。

フレームワーク・ビルド工程なしの素の HTML/CSS/JS で、この `mono/` ディレクトリだけで完結します。

## 機能

- **登録** — カメラ撮影またはアルバムからの選択 + 名前 / カテゴリ / 取得日 / バーコード / メモ。
  カテゴリ欄の下にはマスタ全カテゴリのチップが並び、タップで入力欄に反映されます。
  入力途中の内容はタブを切り替えても保持されます(登録・更新の完了、または編集のキャンセルまで)
- **カテゴリ管理** — 一覧の「カテゴリ編集」チップからダイアログを開き、カテゴリの追加 / 名前変更 / 削除が
  できます。名前変更は該当する全持ち物(手放し済みを含む)のカテゴリに一括反映され、削除した
  カテゴリのアイテムは「未分類」になります。マスタは端末内(localStorage)に保存されます
- **バーコード読取** — ブラウザ標準の `BarcodeDetector` API を優先、非対応環境(iOS Safari 等)では
  zxing-js に自動フォールバック(縦向き・逆さでも読取可。フォールバック時は回転走査で対応)。
  読取後は Open Food Facts で商品名の自動補完を試みる(失敗時は手動入力)
- **一覧** — 写真グリッド / リスト切替、カテゴリチップで絞り込み、取得日・名前・カテゴリで並び替え
- **増減サマリー** — ヘッダーに「今週 +n / −n」を常時表示
- **手放す** — 詳細からワンタップで除外し、方法(処分 / 譲渡 / 売却)と理由を履歴に記録。
  履歴からは「戻す」「完全削除」が可能
- **分析** — 現在の総数 / 今週の増減 / 手放した累計、総アイテム数の12週推移グラフ(SVG)、
  カテゴリ内訳バー(最多カテゴリの偏りをハイライト)
- **オフライン** — Service Worker がアプリシェルをキャッシュ。データは IndexedDB(写真は縮小 JPEG の Blob)に保存

## スマホへのインストール

このリポジトリは GitHub Pages で公開されているため、main にマージされると
**https://tadao-eng.github.io/MIGNON/mono/** で利用できます。

1. スマホのブラウザで上記URLを開く
2. - **Android(Chrome等)**: 画面上部に表示される「インストール」ボタンをタップ
     (表示されない場合はメニュー →「アプリをインストール」)
   - **iPhone / iPad(Safari)**: 共有ボタン →「ホーム画面に追加」
3. ホーム画面のアイコンから起動すると、アドレスバーのない全画面アプリとして動作します

一度開けばアプリシェルはキャッシュされるため、機内モードでも一覧・履歴・分析を閲覧できます。
データは端末内(IndexedDB)にのみ保存され、サーバには送信されません。

## ローカルでの動作確認

カメラと Service Worker は **HTTPS または localhost** でのみ動作します。`file://` では動きません。

```sh
cd mono
python3 -m http.server 8000
# → http://localhost:8000 を開く
```

スマホ実機で試す場合は HTTPS が必要です。GitHub Pages に置くか、開発中なら
`npx serve` + トンネル(`ngrok http 8000` 等)や、同一LANなら Chrome の
`chrome://flags/#unsafely-treat-insecure-origin-as-secure` で開発機の IP を許可する方法があります。

ホーム画面への追加: モバイルブラウザでページを開き「ホーム画面に追加」を選ぶと
スタンドアロンのアプリとして起動します。

## ファイル構成

| ファイル | 役割 |
|---|---|
| `index.html` | 画面(一覧 / 追加 / 履歴 / 分析 + 各ダイアログ) |
| `style.css` | モノトーン + アクセント1色のミニマルデザイン |
| `app.js` | 画面制御・状態管理・描画 |
| `db.js` | IndexedDB ラッパー(`items` ストア1本、`status` で現役/手放し済みを区別) |
| `categories.js` | カテゴリマスタの管理(localStorageへの保存/取得、デフォルト一覧) |
| `scanner.js` | バーコード読取(BarcodeDetector → zxing-js フォールバック) |
| `ai.js` | 品目自動推定(中継サーバURL/Gemini APIキー設定時は高精度判別、未設定時はブラウザ内AI/CLIP、失敗時はさらにTensorFlow.js MobileNetへフォールバック)+ バーコード商品名検索 |
| `ai-labels.js` | MobileNet(フォールバック用)のImageNetクラス名 → カテゴリ/日本語品名 マッピングルール表 |
| `clip-labels.js` | CLIPが照合する候補ラベル一覧(300件超)→ カテゴリ/日本語品名 マッピング表 |
| `sw.js` | Service Worker(アプリシェルのキャッシュ) |
| `manifest.json` | PWA マニフェスト |
| `proxy/` | 任意設置の中継サーバ(Cloudflare Worker)一式。PWA本体には含まれず、Service Workerのキャッシュ対象外 |

## AI連携について(ブラウザ内AI・無料・端末内完結)

写真からの品目名・カテゴリ自動判別は、既定では CLIP(transformers.js によるゼロショット画像分類)を
使ってブラウザの中だけで完結します。サーバには一切送信されず、料金も発生しません。

- CLIPは「こちらが用意した任意のラベル一覧」(`clip-labels.js` の300件超の品目)と写真を照合する
  方式のため、旧来のMobileNet(ImageNetの1000クラスに固定)より幅広い持ち物を判別できます
  (水筒・リモコン・炊飯器・モバイルバッテリー・イヤホン・充電器・マスクなど、ImageNetに無かった
  日用品も対応)
- モデル(transformers.js + CLIP、数十MB)はサイズが大きいため自動先読みはせず、写真を初めて
  選んだ/「AI分析」を初めて押したタイミングで遅延ダウンロードします。初回のみ数秒〜数十秒の
  待ちが発生する場合があり、その間は「AIを準備中…」と案内します。2回目以降はブラウザの
  キャッシュから読み込まれるためほぼ待ちなしで判別が返ります
- 判別できた候補のスコアが低い(=自信がない)場合は「該当なし」として扱い、無理に品目名を
  当てはめません
- 精度は「ざっくり判別」です。候補ラベルに無い持ち物や紛らわしい写り方では外れることもあるため、
  フォームでいつでも修正できるようにしています
- CLIPのダウンロード・推論に失敗した場合(オフライン・CDN不達・古い端末でのWebGPU/WASM非対応
  など)は、従来の TensorFlow.js + MobileNet(ImageNetの1000クラス分類、`ai-labels.js` のルール表で
  変換)に自動的にフォールバックします。写真選択時の自動判別は、この経路も含めいかなる失敗でも
  静かに手動入力へフォールバックします(エラー表示はしません)
- 写真プレビュー下の「AI分析」ボタンから、いつでも手動で再分析できます。この場合は明示的な操作
  であるため、入力済みの名前・カテゴリも推定結果で上書きされます(自動判別は空欄のみ埋めます)。
  手動分析では失敗理由を区別して表示します: CLIP・MobileNetいずれもモデルの読み込み/推論自体に
  失敗した場合は「AIモデルを読み込めませんでした」、判別はできたが候補ラベルに該当が無い場合は
  近い候補名と確度を添えて「判別できませんでした(近い候補: ○○ ○%)」と表示します

### Gemini モード(任意・ユーザー自身のAPIキー使用)

「分析」タブの「AI設定」で自分の Gemini APIキーを保存すると、以降の判別は Google の
Gemini API(高精度・マルチモーダル)に自動的に切り替わります。

- キーは端末内(localStorage)にのみ保存され、サーバやこのアプリの開発者には送信されません。
  写真の判別を行うタイミングでのみ、その写真と判定用プロンプトが Google に送信されます
- APIキーは https://aistudio.google.com/apikey で無料発行できます。無料枠での利用には
  Google の利用規約が適用され、入力内容がモデル品質改善に利用される場合がある点に注意してください
  (詳細は Google AI Studio / Gemini API の利用規約を確認してください)
- キーが無効・期限切れ等の場合は「Gemini APIキーが無効のようです」と表示され、キーを見直すまで
  判別は行われません(自動ではブラウザ内AIにフォールバックしません)
- キーが未設定、またはネットワークエラー・タイムアウトなど認証エラー以外の理由でGemini APIに
  到達できなかった場合は、上記のブラウザ内AI(MobileNet)に自動的にフォールバックします
- 「削除」ボタンでキーを消すと、以降はブラウザ内AIでの判別に戻ります

#### 中継サーバ方式(推奨・APIキーを端末に置かない構成)

Gemini APIキーをブラウザ(端末)にすら保存したくない場合は、自分専用の中継サーバ
(Cloudflare Workers・無料枠で運用可能)を経由する方式が使えます。キーは Worker 側にのみ
保存され、MONO のブラウザ画面には一切残りません。

「分析」タブの「AI設定」にある「中継サーバURL」に、自分でデプロイした Worker のURLを設定すると、
以降の判別は中継サーバ経由(中継サーバ > 直接キー > 端末内AI の優先順位)に切り替わります。
設置手順は `mono/proxy/README.md` を参照してください。中継サーバを使う場合、ブラウザ側の
「Gemini APIキー」欄は不要になるため削除して構いません。

より高精度な判定に差し替えたい場合は、`ai.js` の `classifyWithGemini(photoBlob)` や
`suggestItemInfo(photoBlob)` / `analyzePhoto(photoBlob)` の中身を編集してください
(app.js は各関数の戻り値の形しか見ないため、差し替えは局所的です)。CLIPが照合する候補ラベルは
`clip-labels.js` の `CLIP_LABELS` に分離してあるので、判別できる品目を増やしたい場合はそちらに
`{ en, name, category }` を追加するだけで完結します(MobileNetフォールバック側のマッピングは
`ai-labels.js` の `LABEL_RULES` に分離したままです)。

データ移行が必要になった場合は `db.js` の `DB_VERSION` を上げ、`onupgradeneeded` で対応してください。
クラウド同期を足す場合も、ストレージ層が `db.js` に閉じているため差し替えは局所的です。
