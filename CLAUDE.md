# 開発の基盤ルール(CLAUDE.md)

このリポジトリで作業するClaude(モデル問わず)への恒常的な指示。
オーナー(私)は非エンジニア寄りなので、専門用語は軽く説明を添えること。

## 0. コミュニケーション

- 日本語で。敬語は不要。一人称に「俺」は使わず、中性的または柔らかい口調で
- 結論から先に。長い報告は「何が変わったか → 私がやること → 補足」の順
- 私への質問は選択肢つきでまとめて。作業を止めてまで細かい確認はしない

## 1. モデルの使い分け(コスト方針)

高価なモデル(Fable/Opus級)の消費を節約する。役割分担は固定:

| 役割 | 担当 | 内容 |
|---|---|---|
| 頭脳 | セッション本体(Fable等) | 設計判断、仕様書作成、成果物レビュー、コミット/PR/マージ、難しいバグの診断 |
| 手 | Sonnetのサブエージェント | コーディング、テスト作成・実行、ドキュメント更新、機械的な拡充作業 |
| 雑務 | Haikuのサブエージェント | 単純な一括置換や調査など、判断が要らない作業 |

- 実装作業は原則サブエージェント(Agentツール、model: "sonnet")に委任する
- ただし、数行で終わる修正まで委任するのは逆に無駄。その場で直してよい
- そもそも定型作業しかない依頼なら、セッション自体をSonnetで始めるよう私に提案してよい

## 2. サブエージェントへの委任の型

委任プロンプトには必ず含める:

1. **先に読むファイル**の指定(現状把握なしに書かせない)
2. **確定済みの仕様**(サブエージェントに設計判断をさせない。迷う余地を残さない)
3. **git commit / push の禁止**(コミットは親がレビュー後に行う)
4. **検証の具体的手順**(下記§4)と、その合否の報告義務
5. 報告フォーマット: 変更ファイルと要約 / 検証結果 / 仕様から逸脱した点・懸念

サブエージェントの報告は鵜呑みにせず、親が差分(git diff)の要所を必ず確認してからコミットする。

## 3. Gitの規律

- 作業は指定ブランチで。**mainへ直接pushしない**
- PRのマージは私が「マージまで任せる」と言った場合のみ実行してよい。一度その指示を
  出した後の同種の作業は、同じ進め方(PR作成→マージ→デプロイ確認→報告)を継続してよい
- マージ済みブランチに続きの作業をするときは、最新mainから同名ブランチを作り直す
  (`git fetch origin main && git checkout -B <branch> origin/main`)。
  マージ済み履歴しか無い場合の `push --force-with-lease` は可
- コミットメッセージは英語で「何を・なぜ」を書く。検証結果も一行添える

## 4. 検証の基準(「動くはず」は禁止)

- コード変更は必ず機械で検証してからコミットする:
  - 構文チェック(node --check 等、言語相応のもの)
  - 可能ならヘッドレスブラウザ等でのE2E(主要フロー一式)
  - 外部API・外部リソースはスタブして、成功系だけでなく**失敗系**(通信断・認証エラー・
    上限超過)も通す
- サンドボックスの制約で検証できなかった部分は、報告で正直に「未検証」と明記する
- 公開(デプロイ)後は、デプロイの成否まで確認して報告する

## 5. お金の方針(最重要)

- **無料で完結させるのが基本**。無料枠のあるサービスを優先する
  (GitHub Pages / Cloudflare Workers / Gemini API無料枠 など)
- 課金が発生しうる選択肢を出すときは、必ず「無料の代替案」と「漏洩・超過時の被害上限」を
  併記して私に選ばせる。**私の明示的な同意なしに課金要素を導入しない**
- 「上限を超えたら止まる(課金されない)」構成を「上限を超えたら課金される」構成より常に優先

## 6. セキュリティの方針

- APIキー・トークン・個人情報を**リポジトリに絶対に入れない**(コード、README、コミット
  メッセージ、PR本文すべて)。設定値はユーザーの端末(localStorage等)か、サーバ側の
  Secretに置く設計にする
- 外部にデータ(写真・入力内容など)を送信する機能は、実装前に「何が・いつ・どこへ送られるか」を
  説明して同意を取る
- キーを端末に置く簡易方式を採る場合は、被害上限(無料キーなら枠の消費のみ、等)を説明する。
  課金キーを扱う段になったら、キーをサーバ側に隔離する構成(中継サーバ等)へ移行を提案する

## 7. 進め方のリズム

1. 私の要望を受けたら、方針と選択肢を短く示す(必要なら推奨を1つ明示)
2. 方針が決まったら: 仕様確定(親)→ 実装・検証(Sonnet)→ レビュー(親)→
   コミット/プッシュ → PR → マージ(許可済みの場合)→ デプロイ確認 → 報告
3. 報告には「私が次にやること(あれば)」と「試し方」を必ず書く
4. 未完のまま止まった場合(使用量上限等)は、次の機会に正直に報告してから再開する
